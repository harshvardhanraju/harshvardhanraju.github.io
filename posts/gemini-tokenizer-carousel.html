How Google engineered the most carefully designed tokenizer in production — and why every decision has a specific problem it's solving
Four decisions. Four failure modes fixed.
256K vocab · Unigram LM · Split-digit tokenization · Byte fallback. Each one accepted a cost upfront rather than push the problem into the model.

Decision 2 · Algorithm
Unigram LM, not BPE
BPE (OpenAI · Meta · Qwen)
DirectionMerge up from chars
CriterionPair frequency
ObjectiveGreedy heuristic
TokenizationDeterministic only
BiasHigh-freq lang wins vocab
Unigram LM (Google)
DirectionPrune down from large vocab
CriterionMin log-likelihood loss
ObjectiveProper probabilistic
TokenizationProbabilistic (sampling)
BiasPrincipled coverage
Unigram LM's pruning keeps tokens that are essential for encoding the corpus — not just frequent ones. A rare Telugu morpheme that appears in many word forms stays in the vocab because removing it hurts Telugu log-likelihood, even if it barely affects English. BPE can't make this distinction.

Split-digit doesn't make arithmetic easier — it makes it possible. The model can now learn that "position N+2 from the start of a number = hundreds place" because that relationship is stable. With BPE merges it's random.

Decision 4 · Completeness
Zero unknown tokens — ever
Without byte fallback
Novel Unicode characters → [UNK] token.
OCR errors → [UNK].
Rare scripts → [UNK].

Model loses all information. No recovery.
Gemini: byte_fallback=True
Every byte sequence representable.
Novel scripts → individual byte tokens.
Garbled input → bytes, not garbage.

256 byte tokens always available.
SentencePiece training config (key flags)
model_type='unigram'     # not BPE
vocab_size=256000       # largest in production
split_digits=True       # arithmetic fix
byte_fallback=True      # zero UNK tokens

The Deployment Decision
One vocab. Every Google model.
Gemma 2
Same 256K tokenizer. Same vocab IDs.
Gemma 3
Same 256K tokenizer. Same vocab IDs.
Gemini 1.5
Same 256K tokenizer. Same vocab IDs.
Gemini 2.0
Same 256K tokenizer. Same vocab IDs.
Why lock it? Every tokenizer change breaks fine-tunes, embeddings, RLHF pipelines, and retrieval systems. 256K front-loads the vocabulary budget so you never need to change it. Meta learned this the hard way — LLaMA 2→3 tokenizer change forced a full ecosystem rebuild.
